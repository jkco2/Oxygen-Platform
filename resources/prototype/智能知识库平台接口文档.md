# Oxygen 智能知识库平台接口文档

## 权限模型

### 两层权限

本项目采用“两层权限”：

| 权限层级 | 数据来源 | 用途 |
| --- | --- | --- |
| 系统级权限 | `User -> Role -> Permission` | 控制用户管理、角色管理、创建空间、后台管理 |
| 空间级权限 | `SpaceMember.space_role` | 控制某个知识空间内的查看、编辑、成员管理 |

`User -> Role -> Permission` 管系统级权限，`SpaceMember.spaceRole` 管空间内权限。

### 系统级角色

| 角色编码 | 说明 |
| --- | --- |
| `SYSTEM_ADMIN` | 系统管理员，拥有全站管理权限 |
| `USER_ADMIN` | 用户管理员，可管理用户和角色，扩展 |
| `MEMBER` | 普通注册用户 |

V1 实现 `SYSTEM_ADMIN` 和 `MEMBER`。

### 空间级角色

| 空间角色 | 说明 |
| --- | --- |
| `OWNER` | 空间拥有者，可删除空间、转让空间、管理成员 |
| `ADMIN` | 空间管理员，可管理成员、知识库、文档、条目 |
| `EDITOR` | 编辑者，可创建和编辑知识条目、上传文档 |
| `VIEWER` | 查看者，只能查看知识和向 Agent 提问 |

### Agent 权限原则

Agent 必须先做权限过滤，再检索知识。

正确流程：

```text
用户提问
  -> 校验用户是否能访问 spaceId / kbId
  -> 查询用户可访问的知识库 ID 列表
  -> 只在这些知识库的 chunk 中检索
  -> 调用大模型生成回答
  -> 保存消息和引用来源
```

禁止流程：

```text
先全库检索
  -> 再根据权限隐藏结果
```

这种方式容易造成敏感知识泄露。

### 枚举和可见性约定

接口文档里的 `visibility` 表示“资源对谁可见”，不是编辑权限。

本项目采用以下约定：

```text
数据库：VARCHAR(32)，保存枚举名称，如 PRIVATE、PUBLIC
Java DTO / Entity：使用 enum
前端 JSON：传字符串，如 "PRIVATE"
```

也就是说，不在 Java 代码里到处手写字符串：

```java
// 错误写法
private String visibility;
```

使用枚举：

```java
public enum SpaceVisibility {
    PRIVATE,
    PUBLIC
}
```

```java
private SpaceVisibility visibility;
```

知识空间可见性：

| 枚举值 | 说明 |
| --- | --- |
| `PRIVATE` | 私有空间，只有空间成员和系统管理员可见 |
| `PUBLIC` | 公开空间，所有登录用户可见 |

知识库可见性：

| 枚举值 | 说明 |
| --- | --- |
| `PRIVATE` | 私有知识库，仅空间内有权限的成员可见 |
| `SPACE` | 空间内可见，空间成员可见 |
| `PUBLIC` | 公开知识库，所有登录用户可见 |

空间角色也使用枚举：

```java
public enum SpaceRole {
    OWNER,
    ADMIN,
    EDITOR,
    VIEWER
}
```

注意：

- `visibility` 只控制“能不能看到资源入口”。
- `SpaceMember.spaceRole` 控制“能不能编辑、管理成员、删除空间”。
- 用户能看到公开空间，不代表用户能编辑公开空间。
- MySQL 不使用数据库原生 `ENUM` 类型，使用 `VARCHAR(32)` 更方便扩展。

## 通用接口规范

### 基础路径

```text
/api
```

### 路径命名约定

- Controller 按资源拆分，使用复数资源名作为基础路径。
- 知识空间使用 `/api/knowledge-spaces`，不使用 `/api/spaces`，避免和通用空间概念混淆。
- 父子资源可以在 Controller 类级 `@RequestMapping` 中声明父级路径参数，例如 `@RequestMapping("/api/knowledge-spaces/{spaceId}/members")`。
- 父级归属由路径参数表达时，不再放到 DTO 中，例如 `SpaceMemberAddDTO` 不包含 `spaceId`。
- 修改、删除、详情接口使用当前资源 ID，例如 `PUT /api/knowledge-bases/{kbId}`。
- 状态、角色、密码等子资源接口可以使用 `/{id}/status`、`/{id}/roles`、`/{id}/password`，不强制把路径参数放在最后。

### 认证方式

除注册、登录、公开访问接口外，所有接口都需要携带 JWT。

```http
Authorization: Bearer <accessToken>
```

### 通用响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 分页响应结构

分页查询内部使用 `PageHelper`，但接口不要直接返回 `PageInfo`。项目中定义统一分页返回对象 `PageResult<T>`，只暴露前端真正需要的字段。

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "records": [],
    "total": 100,
    "page": 1,
    "size": 10,
    "pages": 10
  }
}
```

类定义：

```java
package com.ethercraft.result;

import lombok.AllArgsConstructor;
import lombok.Data;

import java.util.List;

@Data
@AllArgsConstructor
public class PageResult<T> {
    private List<T> records;
    private long total;
    private int page;
    private int size;
    private int pages;
}
```

使用约定：

- `PageHelper` 负责拦截 MyBatis 查询并生成分页 SQL。
- `PageInfo<T>` 只在 Service 内部使用。
- Controller 统一返回 `Result<PageResult<T>>`。
- 不直接把 `PageInfo<T>` 返回给前端，因为它包含很多前端用不到的字段。

### 通用错误码

| code | 说明 |
| --- | --- |
| `0` | 成功 |
| `400` | 请求参数错误 |
| `401` | 未登录或 Token 无效 |
| `403` | 无权限 |
| `404` | 资源不存在 |
| `409` | 数据冲突，如用户名重复 |
| `500` | 系统异常 |

## 认证接口

### 用户注册

```http
POST /api/auth/register
```

请求体：

```json
{
  "username": "zhangsan",
  "password": "123456",
  "nickname": "张三",
  "email": "zhangsan@example.com"
}
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "userId": 1
  }
}
```

说明：

- 用户名唯一。
- 密码必须使用 BCrypt 加密存储。
- `nickname` 必填，前端不传时，后端默认使用 `username`。
- `email` 选填。
- 现有代码中的 `UserRegisterDTO` 不包含 `phone`、`avatar`，这两个字段放到用户资料修改接口。
- `email` 建唯一索引时，注册时校验重复。
- 注册成功后默认分配 `MEMBER` 角色。

### 用户登录

```http
POST /api/auth/login
```

请求体：

```json
{
  "username": "zhangsan",
  "password": "123456"
}
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "accessToken": "jwt-token",
    "tokenType": "Bearer",
```

- 现有代码使用 `TokenVO` 作为登录响应对象。
- 前端请求时使用 `Authorization: Bearer <accessToken>`。

```http

```

```text

```

查询参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `page` | Integer | 否 | 页码，默认 1 |

| `keyword` | String | 否 | 用户名、昵称、邮箱、手机号关键字 |
| `status` | UserStatus | 否 | 用户状态：`DISABLED`、`ACTIVE` |

说明：

- `UserPageQueryDTO.status` 使用 `UserStatus`。

响应数据：

```json
{
  "records": [
    {
      "id": 1,
      "username": "zhangsan",
      "nickname": "张三",
      "email": "zhangsan@example.com",
      "phone": "13800138000",
      "avatar": "https://example.com/avatar.png",
      "status": "ACTIVE",
      "roles": ["MEMBER"],
      "createTime": "2026-06-27 05:00:00",
      "lastLoginTime": "2026-06-27 05:10:00"
    }
  ],
  "total": 1,
  "page": 1,
  "size": 10,
  "pages": 1
}
```

对应对象：

- 查询参数使用 `UserPageQueryDTO`。
- 单条用户记录使用 `UserPageVO`。
- 接口返回类型使用 `Result<PageResult<UserPageVO>>`。
- Service 内部使用 `PageHelper.startPage(page, size)` 和 `PageInfo<UserPageVO>`，最后转换为 `PageResult<UserPageVO>`。

## DTO 和 VO 明细

本节用于约束后端类设计，避免一个 DTO 被多个不同场景复用导致字段混乱。

### 用户相关 DTO

#### UserRegisterDTO

用于用户自行注册。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `username` | String | 是 | 登录账号，唯一 |
| `password` | String | 是 | 原始密码 |
| `nickname` | String | 是 | 昵称 |
| `email` | String | 否 | 邮箱 |

不包含：

- `id`
- `status`
- `roleIds`
- `phone`
- `avatar`
- `createTime`
- `updateTime`

#### UserLoginDTO

用于登录。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `username` | String | 是 | 登录账号 |
| `password` | String | 是 | 登录密码 |

#### UserCreateDTO

用于管理员后台创建用户。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `username` | String | 是 | 登录账号 |
| `password` | String | 是 | 初始密码 |
| `nickname` | String | 是 | 昵称 |
| `email` | String | 否 | 邮箱 |
| `phone` | String | 否 | 手机号 |
| `avatar` | String | 否 | 头像地址 |
| `status` | UserStatus | 否 | 用户状态：`DISABLED`、`ACTIVE` |
| `roleIds` | List<Long> | 否 | 初始角色，不传默认 `MEMBER` |

说明：

- 现有代码字段名是 `roleIds`，注意不要写成 `roldIds`。
- `UserCreateDTO.status` 不传时，Service 默认设置为 `ACTIVE`。

#### UserUpdateDTO

用于修改用户基础资料。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `nickname` | String | 是 | 昵称 |
| `email` | String | 否 | 邮箱 |
| `phone` | String | 否 | 手机号 |
| `avatar` | String | 否 | 头像地址 |

不包含：

- `id`：来自路径参数 `{userId}`
- `username`：账号不允许随意修改
- `password`：改密码单独接口
- `status`：启用禁用单独接口
- `roleIds`：分配角色单独接口

#### UserStatusUpdateDTO

用于启用或禁用用户。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `status` | UserStatus | 是 | `DISABLED`、`ACTIVE` |

#### UserRoleAssignDTO

用于给用户覆盖式分配角色。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `roleIds` | List<Long> | 是 | 用户最终拥有的角色 ID 列表 |

#### UserPasswordResetDTO

用于后台管理员重置用户密码。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `newPassword` | String | 是 | 新密码 |

#### PasswordUpdateDTO

用于用户自己修改密码。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `oldPassword` | String | 是 | 旧密码 |
| `newPassword` | String | 是 | 新密码 |

#### UserPageQueryDTO

用于后台分页查询用户。该 DTO 接收 GET 查询参数，不使用 `@RequestBody`。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `page` | Integer | 否 | 页码，默认 1 |
| `size` | Integer | 否 | 每页数量，默认 10 |
| `keyword` | String | 否 | 用户名、昵称、邮箱、手机号关键字 |
| `status` | UserStatus | 否 | 用户状态：`DISABLED`、`ACTIVE` |

默认值：

```java
private int page = 1;
private int size = 10;
```

说明：

- 使用 `PageHelper` 时不需要在 DTO 中提供 `offset`。
- `page` 小于 1 时后端修正为 1。
- `size` 限制最大值，比如不超过 100。
- `UserPageQueryDTO.status` 使用 `UserStatus`，不传表示不过滤状态。

### 用户相关 VO

#### TokenVO

用于登录响应。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `accessToken` | String | JWT |
| `tokenType` | String | Token 类型，固定为 `Bearer` |
| `expiresIn` | Long | 过期时间，现有代码含义是毫秒数 |
| `userLoginVO` | UserLoginVO | 登录用户基础信息 |

说明：

- 现有代码已使用 `TokenVO`，不要再写成 `LoginVO`。
- 如果现阶段不想返回 `tokenType`、`expiresIn`，从 `TokenVO` 中删掉；保留的话就要在 Service 中赋值。

#### UserLoginVO

用于嵌套在 `TokenVO` 中，返回登录用户基础信息。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | Long | 用户 ID |
| `username` | String | 登录账号 |
| `nickname` | String | 昵称 |

#### UserMeVO

用于 `/api/auth/me` 返回当前登录用户信息。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | Long | 用户 ID |
| `username` | String | 登录账号 |
| `nickname` | String | 昵称 |
| `roles` | List<Role> | 当前用户拥有的角色对象列表 |
| `permissions` | List<Permission> | 当前用户拥有的权限对象列表 |

不允许返回：

- `password`

说明：

- 现有代码直接返回 `Role` 和 `Permission` Entity，能用，但会把实体字段完整暴露给前端。
- 后续可新增 `RoleVO`、`PermissionVO`，或只返回 `List<String>` 的角色编码、权限编码。

#### UserPageVO

用于后台分页查询用户列表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | Long | 用户 ID |
| `username` | String | 登录账号 |
| `nickname` | String | 昵称 |
| `email` | String | 邮箱 |
| `phone` | String | 手机号 |
| `avatar` | String | 头像 |
| `status` | UserStatus | 用户状态：`DISABLED`、`ACTIVE` |
| `roles` | List<String> | 角色编码列表 |
| `createTime` | LocalDateTime | 创建时间 |
| `lastLoginTime` | LocalDateTime | 最后登录时间 |

说明：

- 用户列表页不返回 `permissions`，避免每条用户都查权限导致数据过重。
- 查看某个用户的完整权限时，使用用户详情接口。

#### UserRoleCodeVO

用于批量查询用户角色编码后在 Service 中组装 `UserPageVO.roles`，一般不直接返回给前端。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `userId` | Long | 用户 ID |
| `roleCode` | String | 角色编码 |

使用场景：

```text
分页查出 users
  -> 提取 userIds
  -> 批量查询 UserRoleCodeVO
  -> 按 userId 分组
  -> 填充 UserPageVO.roles
```

### 通用返回对象

#### PageResult<T>

用于统一分页接口返回。`PageHelper` 的 `PageInfo<T>` 只在 Service 内部使用，Controller 返回 `Result<PageResult<T>>`。

| 字段 | 类型 | 说明 |
| --- | --- | --- |

| `page` | int | 当前页码 |
| `size` | int | 每页数量 |
| `pages` | int | 总页数 |

定义：

```java
@Data
@AllArgsConstructor
public class PageResult<T> {
    private List<T> records;
    private long total;
    private int page;
    private int size;
    private int pages;
}
```

## 权限注解方式

系统级权限直接使用 Spring Security 注解：

```java
@PreAuthorize("hasAuthority('user:view')")
@GetMapping("/api/admin/users")
public Result<PageResult<UserPageVO>> pageUsers(UserPageQueryDTO queryDTO) {
    return Result.success(userService.pageUsers(queryDTO));
}
```

空间级权限封装业务方法，不全部写在注解里：

```java
    SpaceMember member = spaceMemberService.getByUserIdAndSpaceId(userId, spaceId);
    if (member == null || !allowedRoles.contains(member.getSpaceRole())) {
        throw new BizException(403, "无空间操作权限");
    }
}
```
